If no bad rows appear, then your counts are raw integer style and okay for DESeq2.

Step 3 — Check whether one read was assigned multiple times
This is very important.
Go to your genes folder:

Now inspect one read-to-gene file:
head S10_R1.read_gene.tsv

You should see something like:

Now check whether the same read ID appears more than once in the same file:
cut -f1 S10_R1.read_gene.tsv | sort | uniq -d | head

Interpretation:
if this prints nothing, good
if it prints read IDs, then some reads are assigned multiple times

Now count how many duplicated read IDs there are:
cut -f1 S10_R1.read_gene.tsv | sort | uniq -d | wc -l

Do the same for R2:
cut -f1 S10_R2.read_gene.tsv | sort | uniq -d | wc -l

Now check all files together:
for f in *_R1.read_gene.tsv *_R2.read_gene.tsv
do
  dups=$(cut -f1 "$f" | sort | uniq -d | wc -l)
  echo "$f : $dups duplicated read IDs"
done
This is the key result
if duplicated read IDs are 0 or very few, fine
if duplicated read IDs are many, then one read may be counted for multiple genes, and we need to fix that before DESeq2

Step 4 — Check whether counts are extremely low for too many genes
Go back to counts:
cd ~/Ramya/Mobile/counts

Check one file:
awk '$2 < 5' S10.gene_readcounts.tsv | wc -l
awk '$2 >= 5' S10.gene_readcounts.tsv | wc -l

Now check all files:

Interpretation:
If almost all genes are under 5, your data is very sparse.
That does not mean impossible, but we will need stronger filtering before DESeq2.

Step 5 — Check total counts per sample
This helps us see whether one sample has too many or too few Lagenaria-supporting reads.

Run:

  total=$(awk '{sum+=$2} END{print sum}' "$f")

We compare totals across samples.
Some variation is okay.
But if one sample has, for example:

if nothing prints, good
if rows print, then file format is inconsistent

summary of why correction and rerun of pipeline needed:

Before proceeding with normalization and differential expression analysis, several quality checks were performed on the generated gene read count files to ensure they were suitable for downstream analysis. First, the count files were inspected to confirm that they contained raw integer counts for each Lagenaria gene across all samples. While the file format itself was correct, further checks revealed a critical issue during the read-to-gene assignment stage. When examining the read_gene.tsv files, many duplicated read IDs were detected, indicating that the same unmapped read had been assigned to multiple Lagenaria genes. This occurred during the bedtools intersect step, where a single BLAST hit overlapped more than one gene region in the genome annotation, causing the read to be counted multiple times. As a result, gene-level read counts were artificially inflated and could potentially bias normalization and DEG results. Because accurate quantification requires that each read contribute only once, the pipeline was rerun from the gene-assignment stage onward with an additional filtering step to ensure that each read ID was linked to only one gene. The corrected pipeline removed duplicated read assignments, regenerated the gene lists and gene read count files, and produced cleaner count matrices suitable for normalization and differential expression analysis.
